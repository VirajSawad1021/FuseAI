Config is better in a .env file because it keeps secrets out of the codebase and makes the application easier to move between environments. If the password changes, only the environment changes, not the source code. That is safer and more maintainable. It also makes it easier to give different values for local development, testing, and production without editing Python files.

Treating PostgreSQL as a separate backing service is useful because the application does not need to own the database process. The API can focus on business logic while the database can be started, stopped, upgraded, or replaced independently. With an ORM layer, the code also stays cleaner if the storage engine changes later. That separation makes the system easier to reason about and easier to test in isolation.

Docker makes development and production similar because the same container image and startup process can run in both places. That reduces configuration drift and avoids the common problem of code working locally but failing in deployment. The database, schema initialization, and runtime behavior all stay closer to what will happen in production, which makes testing more reliable. In practice, this means the same seed process, the same dependencies, and the same startup flow can be reused with fewer surprises.

Together, these three ideas make the project easier to maintain. Configuration stays outside the code, the database remains a separate service, and Docker keeps the environment consistent. That combination is what makes the assignment closer to how a real application would be built and deployed.
